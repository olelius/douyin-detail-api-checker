# Node.js detail API 完整实现计划

> **给后续子智能体：** 本计划只用于指导实现。执行时请逐项勾选，不要回退或覆盖他人无关改动。当前项目不是正常 Git 仓库，不要提交 commit。所有新增或更新文件统一使用 UTF-8 无 BOM，所有文档、注释、日志、Excel 列名使用中文。

**目标：** 在 `methods/node_detail_api` 内完成 Node.js + Playwright 捕获抖音 detail API 的链接有效性检测工具，支持从 Excel 读取链接、有限并发检测、输出中文结果表格，并保留 DOM 文本兜底。

**架构：** `main.js` 负责命令行入口、默认路径、并发调度、进度日志和汇总统计；`detector.js` 负责浏览器启动、页面打开、detail API 响应监听、DOM 兜底和错误归一化；`classifier.js` 负责把 detail API JSON、最终 URL、DOM 文本归类为存活、失效或待确认；`excel.js` 负责 Excel 读取、链接列自动识别和结果写出。文档与启动脚本放在 Node 目录内，旧 Python 方案只作为样例和对照，不修改。

**技术栈：** Node.js v22.13.0、ESM、Playwright、exceljs、p-limit、node:test、系统 Chrome/Edge。

### 任务 2：实现 classifier 的可测试判定规则

**目标：** 先实现纯函数判定逻辑，保证没有真实抖音接口也能单测核心分类规则。

**修改文件：**
- `methods/node_detail_api/src/classifier.js`
- `methods/node_detail_api/test/classifier.test.js`

**执行要点：**
- 导出 `classifyDetailResult(result)`，输入统一使用 detector 返回对象：
  - `originalUrl`
  - `finalUrl`
  - `detailJson`
  - `fallbackText`
  - `apiUrl`
  - `error`
- 输出统一包含：
  - `status`: `存活`、`失效`、`待确认`
  - `contentType`: `视频`、`图文`、`未知`
  - `reason`: 中文判定原因
  - `basis`: `detail_api`、`dom_text`、`url_redirect`、`error`、`heuristic`
- detail API 优先，DOM 只做兜底。
- 结构化 JSON 启发式规则必须覆盖：
  - JSON 中存在 `aweme_detail`、`aweme_id`、`desc`、`statistics`、`images`、`video` 等有效内容时判定为 `存活`。
  - JSON 中 `aweme_detail`、`item_list`、`data` 等核心内容为 `null`、空数组或缺失，同时存在错误码、状态码、错误信息时判定为 `失效` 或 `待确认`。
  - `images`、`image_post_info`、最终 URL 含 `/note/` 时优先识别为 `图文`。
  - `video`、最终 URL 含 `/video/` 或 `/share/video/` 时识别为 `视频`。
  - 原始 `/video/` 最终跳到 `/note/` 且有存活证据时，判定为 `存活` + `图文`，原因说明“视频链接跳转图文”。
- DOM 兜底关键词：
  - 失效：`你要观看的视频不存在`、`你要观看的图文不存在`、`抱歉，作品不见了`
  - 存活：`点赞`、`评论`、`发布时间`、`关注`
- 无 API、无 DOM 明确信息、或出现风控/登录/验证码文案时，判定 `待确认`。

**验证命令：**
- `node --test test/classifier.test.js`
  - 期望：覆盖视频存活、视频失效、图文存活、图文失效、`/video/` 跳 `/note/`、DOM 兜底存活、DOM 兜底失效、风控待确认。
- `node --check src/classifier.js`
  - 期望：无语法错误。

  - 如果无表头匹配，扫描前若干行中含 `douyin.com`、`/video/`、`/note/`、`/share/video/` 的单元格，给出明确错误或自动识别。

### 任务 4：实现 detector 的 Playwright detail API 捕获

**目标：** 使用系统 Chrome/Edge 打开抖音页面，捕获 detail API 响应 JSON，并返回最终 URL、DOM 兜底文本和错误信息。

**修改文件：**
- `methods/node_detail_api/src/detector.js`
- 可选 `methods/node_detail_api/src/config.js`

**执行要点：**
- 使用 Playwright 的 `chromium.launch`。
- 优先使用系统 Chrome：`C:\Program Files\Google\Chrome\Application\chrome.exe`。
- 如果 Chrome 不存在，再尝试常见 Edge 路径：`C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe`、`C:\Program Files\Microsoft\Edge\Application\msedge.exe`。
- 如未找到系统浏览器，提示用户安装 Chrome/Edge 或执行 Playwright 浏览器安装，不要静默失败。
- 设置真实浏览器 UA 和中文地区参数。
- 不使用 `networkidle`。
- 推荐流程：
  - 创建浏览器上下文。
  - 在页面上注册 `response` 监听。
  - 判断 detail API 的 URL 关键词，建议先覆盖 `aweme/detail`、`aweme/v1/web/aweme/detail`、`multi/aweme/detail`、`note`、`detail` 等可配置规则。
  - `page.goto(url, { waitUntil: "domcontentloaded", timeout })`。
  - 等待固定时间 `waitAfterLoadMs`，默认 6000ms。
  - 尝试读取 `document.body.innerText`。
  - 记录 `page.url()` 作为最终链接。
  - 清理 page/context，复用 browser 或 detector 生命周期统一关闭。
- 返回对象建议包含：
  - `originalUrl`
  - `finalUrl`
  - `apiUrl`
  - `detailJson`
  - `fallbackText`
  - `error`
  - `timedOut`
  - `startedAt`
  - `finishedAt`
- 对接口 JSON 解析失败、页面超时、浏览器启动失败分别给出中文错误。
- detector 只负责采集证据，不直接写 Excel；分类交给 classifier。

**验证命令：**
- `node --check src/detector.js`
  - 期望：无语法错误。
- 在网络可用且用户允许真实访问时，小批量验证：
  - `node src/main.js input/测试.xlsx output/检测结果.xlsx --limit 3 --concurrency 1`
  - 期望：能打开页面，输出最终 URL、检测依据；如遇风控，状态为 `待确认` 并在备注记录限制。

## 风险

1. 抖音接口路径、JSON 字段和前端文案可能变化，所以 classifier 必须把 detail API 规则写成可测试、可扩展的启发式，而不是只匹配单一字段。
2. 真实页面可能触发登录、验证码、地区限制或风控，不能把无法访问误判为失效，应输出 `待确认`。
3. `networkidle` 可能早于关键 API 或因长连接不稳定，明确禁止使用。
4. Windows PowerShell 执行策略会导致直接 `npm` 失败，文档和 bat 必须使用 `cmd /c npm ...`。
5. 当前骨架文件存在中文乱码风险，实现前必须统一 UTF-8 无 BOM，否则文档和日志不可维护。
6. 并发过高可能触发风控，默认并发建议为 `1`，仅允许通过参数小幅调整。
7. 旧 Python 方案仍需保留，不要删除或重构 `methods/python_playwright_dom_text`。
