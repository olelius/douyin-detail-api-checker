# UI 服务设计文档

> 本文档只描述后续本地 Web UI 服务的设计方案，不包含业务代码实现。旧 Python 方案 `methods/python_playwright_dom_text` 继续保留，不在本阶段修改。

## 目标

在 `methods/node_detail_api` 现有 Node CLI 基础上，新增本地 Web UI 服务。用户双击 `run.bat` 后启动 `http://localhost:3000`，浏览器首屏直接进入抖音链接有效性检测工作台，支持导入 Excel、配置参数、启动或停止任务、查看进度、统计、日志、结果表格预览，并导出追加检测结果列后的 Excel。

核心检测架构升级为两阶段：

1. 第一阶段使用 Cookie + HTTP detail API 快筛，默认 API 并发 `5`。
2. 第二阶段只对待确认、API 异常、风控疑似项使用 Playwright 页面兜底检测，兜底并发先测试 `2`，如不稳定降为 `1`。

服务应延续现有判定原则：detail API 结构化结果优先，DOM 文本只做兜底；不使用 `networkidle`；输出列名、日志、页面文案和文档均使用中文。

## 非目标

- 不做公网服务，不引入账号体系、远程数据库或多人协作能力。
- 不做 landing page，首屏就是工具工作台。
- 不修改 `methods/python_playwright_dom_text` 旧 Python 方案。
- 不删除现有 CLI；Web UI 应复用或封装现有能力，并保留命令行可用性。
- 不在 UI 中展示或保存敏感 Cookie 明文；不提供绕过验证码、登录或平台风控的功能。
- 不追求高并发压测，优先稳定、可控、可解释。
- 不在设计阶段创建 `server.js`、`public`、前端组件或安装新依赖。

## 目录结构建议

后续实现建议在 `methods/node_detail_api` 内按职责分层，避免把服务、队列、检测和 UI 混在一个大文件中。

```text
methods/node_detail_api/
  src/
    main.js                  # 保留现有 CLI 入口
    config.js                # 扩展统一配置：端口、并发、超时、限速
    classifier.js            # 继续作为判定规则核心
    excel.js                 # 继续负责 Excel 读写与结果列追加
    detector.js              # 保留现有 Playwright 页面兜底能力
    api-detector.js          # 建议新增：Cookie + HTTP detail API 快筛
    task-manager.js          # 建议新增：任务生命周期、取消、进度、统计
    queue.js                 # 建议新增：两阶段队列、并发、节流
    cookie-store.js          # 建议新增：Cookie 获取、校验、刷新、持久化
    web-server.js            # 建议新增：本地 HTTP 服务与静态资源
  web/
    index.html               # 工作台页面入口
    app.js                   # 前端状态管理、轮询或 SSE、交互
    styles.css               # 安静工作型 UI 样式
  uploads/                   # 导入文件临时目录，建议加入忽略规则
  output/                    # 导出结果目录，沿用现有输出习惯
  docs/
    ui-service-设计文档.md
```

如果后续选择 Vite、React 或其他前端工具链，可将 `web/` 调整为对应结构，但首版建议优先使用轻量静态页面，降低安装依赖和打包复杂度。

## 服务端架构

服务端建议采用本地 Node HTTP 服务，监听 `127.0.0.1:3000` 或 `localhost:3000`。端口被占用时应给出中文错误提示，不静默切换端口，避免用户找不到入口。

建议模块职责：

| 模块 | 职责 |
|------|------|
| `web-server` | 提供静态页面、文件上传、任务创建、任务查询、停止任务、导出下载接口。 |
| `task-manager` | 管理单个或少量本地任务，维护任务状态、进度、统计、日志、结果缓存和取消信号。 |
| `queue` | 执行两阶段检测队列，分别控制 API 快筛并发和 Playwright 兜底并发。 |
| `api-detector` | 使用 Cookie 和 HTTP 请求访问候选 detail API，返回结构化原始证据。 |
| `detector` | 复用现有 Playwright DOM 兜底逻辑，不使用 `networkidle`。 |
| `classifier` | 复用现有分类规则，并补充 HTTP API 返回形态的单测覆盖。 |
| `excel` | 复用现有 workbook 读取和写出策略，保证样式、列宽、其他工作表尽量保留。 |
| `cookie-store` | 管理 Cookie 来源、有效性检查、刷新提示、过期状态和本地存储。 |

接口建议：

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/` | 返回工作台页面。 |
| `GET` | `/api/health` | 返回服务状态、版本、端口、浏览器检测状态。 |
| `POST` | `/api/tasks` | 上传 Excel 并创建任务，参数包含并发、延迟、超时、是否启用兜底等。 |
| `GET` | `/api/tasks/:id` | 查询任务状态、进度、统计和最近日志。 |
| `POST` | `/api/tasks/:id/stop` | 请求停止任务，队列不再拉取新链接，正在执行项尽量自然收尾。 |
| `GET` | `/api/tasks/:id/results` | 分页获取结果表格。 |
| `GET` | `/api/tasks/:id/export` | 下载输出 Excel。 |
| `GET` | `/api/cookie/status` | 查询 Cookie 是否存在、最近刷新时间、可用性摘要。 |
| `POST` | `/api/cookie/refresh` | 触发 Cookie 刷新流程，必要时打开浏览器让用户登录确认。 |

实时进度首版可使用短轮询，后续可升级为 SSE。WebSocket 不是首版必需。

## 前端 UI 架构

UI 定位为本地工作台，不做营销页。首屏应包含工具本体：

- 顶部状态栏：服务状态、Cookie 状态、当前任务状态、导出入口。
- 左侧任务面板：文件选择或拖拽上传、参数控制、开始、停止。
- 中部进度与统计：总数、已处理、存活、失效、待确认、跳过、失败、API 命中、兜底数量。
- 右侧或底部日志区：按时间追加关键事件，可过滤错误、风控、兜底。
- 主表格预览：按行展示序号、原始链接、状态、内容类型、最终链接、备注、检测依据、耗时。

视觉风格应安静、工作型、信息密度适中。按钮、输入、表格、日志和统计应优先服务扫描与操作效率，不做大幅 hero、装饰插画、悬浮卡片堆叠或过度渐变。

参数控件建议：

| 控件 | 默认值 | 说明 |
|------|--------|------|
| API 快筛并发 | `5` | 第一阶段 HTTP detail API 并发。 |
| Playwright 兜底并发 | `2` | 先小批量验证，不稳定降为 `1`。 |
| 单项超时 | `15000ms` | 沿用现有 CLI 默认超时。 |
| 批次间隔 | `1000ms` | 降低触发风控概率。 |
| 是否启用兜底 | 开启 | 待确认、API 异常、风控疑似项进入第二阶段。 |
| 仅检测前 N 条 | 关闭 | 仅测试验证使用；正式检测不应启用。 |

停止按钮语义为“请求停止”，不是强杀。UI 应显示“停止中”，等当前执行项收尾后进入“已停止”。

## 两阶段检测流程

### 阶段一：Cookie + HTTP detail API 快筛

1. 从 Excel 读取链接并识别链接列。
2. 提取作品 ID 和 URL 类型，记录原始 URL。
3. 使用当前 Cookie 和真实浏览器 UA 构造 HTTP 请求。
4. 对候选 detail API 返回 JSON 做结构化解析。
5. 调用 `classifier` 判定存活、失效、内容类型和检测依据。
6. 明确存活或失效的结果直接写入任务结果缓存。
7. 以下项目进入第二阶段：
   - 无法提取作品 ID。
   - HTTP 请求超时、网络异常、解析异常。
   - API 返回登录、验证码、安全验证、访问频繁等风控疑似信号。
   - API 数据结构变化导致分类器无法确认。
   - `/video/`、`/note/` 类型与结果证据冲突。

### 阶段二：Playwright 页面兜底

1. 仅处理第一阶段未确认或异常的链接。
2. 复用现有 Playwright 检测策略：`domcontentloaded` 加固定等待，读取最终 URL 和 `document.body.innerText`。
3. 监听 detail API 响应作为额外证据，但候选 JSON 解析失败只记录为非致命错误。
4. 调用 `classifier` 复判。
5. 如果仍遇到登录、验证码、安全验证、访问频繁等文案，输出 `待确认`，不能误判为失效。

全流程不得使用 `networkidle`。

## Cookie 获取与刷新策略

Cookie 是第一阶段快筛的关键输入，但不能让用户手工复制复杂 Cookie 字符串作为唯一方案。

建议策略：

1. 首次启动时检查本地 Cookie 存储状态。
2. 如果没有 Cookie，UI 显示“需要登录 Cookie”，用户点击“刷新 Cookie”。
3. 服务端打开系统 Chrome/Edge 或 Playwright 持久化上下文，让用户在抖音页面完成登录或确认登录状态。
4. 用户完成后点击“我已完成”，服务端读取当前上下文 Cookie，并写入本地 Cookie 存储。
5. Cookie 存储只用于本地检测，不在 UI 中展示明文。
6. 每次任务开始前执行轻量可用性检查；如果返回登录、验证或权限异常，标记 Cookie 可能失效。
7. 执行过程中连续出现风控疑似项时暂停 API 快筛，提示用户刷新 Cookie 或降低并发。

本地存储建议：

- 文件放在 `methods/node_detail_api/.local/cookies.json` 或系统用户数据目录。
- 文档和 UI 明确提示这是本机本地文件。
- 如实现成本允许，优先使用系统凭据或简单加密；首版至少避免日志和页面输出 Cookie 明文。
- Cookie 刷新不会删除任务结果；刷新后可继续处理未完成项。

## 并发、队列与限速策略

队列应按两阶段分开控制：

| 队列 | 默认并发 | 建议限速 | 说明 |
|------|----------|----------|------|
| API 快筛队列 | `5` | 每批之间保留短延迟和抖动 | 负责大部分链接的快速判定。 |
| Playwright 兜底队列 | `2` | 每个页面打开前至少 `1000ms` 间隔 | 不稳定或风控增加时降为 `1`。 |
| Excel 写出 | `1` | 任务结束或检查点写出 | 避免并发写 workbook。 |

任务取消策略：

- 停止后不再启动新请求。
- 已发出的 HTTP 请求等待超时或自然返回。
- 已打开的 Playwright 页面完成当前读取后关闭。
- 结果文件应保留已完成行，未完成行标记为 `待确认` 或 `跳过`，备注写明“任务停止未检测”。

风控降级策略：

- API 阶段连续命中风控疑似时，将 API 并发从 `5` 降为 `2`，仍持续异常则暂停并提示刷新 Cookie。
- 兜底阶段连续出现验证或访问频繁时，将兜底并发从 `2` 降为 `1`。
- 降级事件必须写入日志，便于复盘。

## 上千条数据处理策略

上千条 Excel 不应一次性把所有详细日志和完整 DOM 文本塞入前端。

建议策略：

- 输入读取后只保留必要字段：行号、原始链接、原始行引用、当前状态。
- 任务结果以行号为 key 更新，前端结果表格分页查询。
- 日志采用环形缓冲，UI 默认只拉取最近 200 条，完整日志可另存。
- DOM 文本只保留命中摘要，不长期保留整页文本。
- 输出 Excel 优先在任务完成后统一写出；长任务可每 100 条或每 5 分钟生成检查点文件。
- 前端表格使用分页或虚拟滚动，避免一次渲染上千行导致卡顿。
- 任务恢复首版不是必需，但检查点文件应能帮助人工追溯。

## 错误处理

错误应分层归一化，避免把可恢复问题变成任务失败。

| 类型 | 处理 |
|------|------|
| Excel 无法读取 | 任务创建失败，提示文件格式、工作表和链接列检查。 |
| 链接为空 | 标记 `跳过`，备注 `链接为空`。 |
| URL 不支持 | 标记 `待确认`，备注说明不属于支持的抖音链接类型。 |
| HTTP 超时或解析失败 | 进入 Playwright 兜底；若兜底仍无证据，标记 `待确认`。 |
| Cookie 失效 | 暂停或降级 API 队列，提示刷新 Cookie。 |
| 登录、验证码、风控 | 标记 `待确认`，不能判定失效。 |
| Playwright 启动失败 | 当前项标记 `待确认`，任务日志提示安装或检查 Chrome/Edge。 |
| 输出 Excel 失败 | 保留内存结果和临时结果，提示关闭已打开的输出文件后重试导出。 |
| 端口 3000 被占用 | 启动失败并提示释放端口，不自动切换。 |

所有错误日志使用中文，包含发生阶段、行号、链接、简短原因和建议操作。候选 API JSON 解析失败属于非致命事件，只能影响该响应证据，不能覆盖 DOM 兜底明确结论。

## Excel 导入导出策略

导入沿用现有 `excel.js` 设计：

- 默认读取第一个工作表。
- 优先识别表头等于 `链接` 的列。
- 其次识别包含 `链接`、`URL`、`url`、`地址` 的表头。
- 表头无法识别时，扫描前若干行包含 `douyin.com`、`/video/`、`/note/`、`/share/video/` 的单元格。

导出要求：

- 基于输入 workbook 追加结果列并另存，不新建空 workbook 复制 value。
- 尽量保留 ExcelJS 可保留的工作表、列宽、样式、公式、超链接、合并单元格。
- 输出列沿用中文：`状态`、`最终链接`、`备注`、`检测时间`、`检测依据`。
- Web UI 可额外在内存结果中展示 `内容类型`、`耗时`、`API 阶段结果`、`兜底阶段结果`，但是否写入 Excel 需要单独规格审查，避免破坏既有用户预期。
- 如果输出文件被 Excel 打开导致写出失败，应提示用户关闭文件后点击“重新导出”。

## run.bat 启动策略

新阶段 `run.bat` 目标从“直接执行 CLI 检测”调整为“启动本地 Web UI 服务”。

建议流程：

1. 切换到 `methods/node_detail_api` 当前目录。
2. 执行 `chcp 65001` 保证中文输出。
3. 检查 Node.js 是否存在。
4. 检查 `node_modules` 是否存在；不存在时提示执行 `cmd /c npm install`。
5. 执行语法检查或服务启动前检查。
6. 启动本地服务，监听 `http://localhost:3000`。
7. 自动打开默认浏览器访问 `http://localhost:3000`。
8. 服务窗口保留，显示启动状态、端口和退出提示。

为保留 CLI 能力，建议新增单独脚本或 npm 命令：

- `cmd /c npm start`：启动 UI 服务。
- `node src/main.js`：继续执行 CLI 检测。
- 如需双击执行 CLI，可另建 `run-cli.bat`，但是否新增由后续实现规格审查决定。

## 测试与验收标准

### 单元测试

- `classifier` 覆盖 HTTP API 快筛返回形态、DOM 兜底、风控待确认、`/video/` 跳 `/note/`。
- `api-detector` 使用 mock HTTP 响应测试 Cookie 注入、超时、异常、JSON 解析失败。
- `queue` 覆盖并发限制、停止任务、风控降级、第二阶段入队规则。
- `excel` 覆盖导入识别、输出列追加、样式和多工作表保留能力。

### 集成测试

- 使用本地 mock detail API，不访问真实抖音网络。
- 上传测试 Excel 后创建任务，验证任务状态从 `等待中` 到 `检测中` 再到 `已完成`。
- 验证 API 明确存活和失效时不进入 Playwright 兜底。
- 验证 API 异常或待确认时进入兜底队列。
- 验证停止任务后不再启动新检测项，已完成结果可导出。

### UI 验收

- 双击 `run.bat` 后能打开 `http://localhost:3000`。
- 首屏直接是工作台，不出现 landing page。
- 能选择或上传 Excel，能看到参数控件和开始按钮。
- 任务执行时能看到进度、统计、日志和表格预览。
- 停止、导出、错误提示都有明确中文反馈。
- 页面在常见桌面宽度下不发生文字重叠；表格长链接可截断、复制或横向滚动。

### 非真实网络验证命令建议

设计和基础实现阶段优先使用：

```bash
cmd /c npm run check
cmd /c npm test
```

涉及 UI 服务时可增加本地 mock 集成测试。真实抖音检测只能在用户明确要求验收时运行，且应小批量、低并发、记录命令。

## 后续子智能体拆分建议

用户要求后续所有代码相关工作必须按“主智能体调度，子智能体开发，规格审查，代码质量审查，有问题打回”的流程执行。建议拆分如下：

1. 规格审查子智能体：审查本文档是否覆盖需求、是否与现有 CLI 冲突、是否存在不可验收的模糊点。
2. 服务骨架子智能体：只实现本地服务启动、健康检查、静态页面托管和 `run.bat` 启动策略，不接入真实检测。
3. 任务队列子智能体：实现任务状态机、队列、并发、停止、日志和 mock 检测。
4. Cookie 与 HTTP API 子智能体：实现 Cookie 获取、存储、刷新状态和 HTTP detail API 快筛，使用 mock 测试优先。
5. Playwright 兜底接入子智能体：复用现有 `detector.js`，只处理第二阶段待确认项，不使用 `networkidle`。
6. 前端工作台子智能体：实现文件选择、参数、开始停止、进度统计、日志、分页表格和导出入口。
7. Excel 导入导出子智能体：将 Web 任务结果接回现有 workbook 输出策略，重点验证保留格式。
8. 代码质量审查子智能体：审查并发、错误处理、敏感信息、测试覆盖和是否误改旧 Python 方案。
9. 验收子智能体：只运行非真实网络测试和本地 UI 验收；真实检测需主智能体确认后再执行。

每个子智能体完成后必须更新 `docs/子智能体处理记录.md`，记录修改文件、验证命令、结果和遗留风险。

## 主要设计结论

- Web UI 是现有 Node CLI 的本地工作台外壳，不替代核心判定规则。
- 第一阶段 API 快筛负责效率，第二阶段 Playwright 兜底负责准确性和异常确认。
- Cookie 管理必须本地化、可刷新、不可在日志或 UI 中泄露明文。
- 并发策略必须可降级，风控疑似不能误判为失效。
- Excel 输出继续以“基于原 workbook 追加结果列并另存”为原则。
- `run.bat` 后续应启动 `http://localhost:3000`，CLI 能力另行保留。
